02 What Basilisk Actually Is

The AVS Lab at the University of Colorado Boulder describes Basilisk as a software framework, not a program. The distinction matters because Basilisk by itself does not simulate anything. Instead, it provides the building blocks from which spacecraft simulations are assembled. The documentation uses a Lego analogy that I think is exactly right: each module is a self-contained brick, and the user's job is to stack and connect them into the spacecraft behavior they need. The modules encapsulate the physics and algorithms; the user wires them together through a message-passing interface and never needs to understand the math inside each module to use it correctly.

The Basilisk module interface: output messages from one module feed into input messages of the next. Connecting these messages is how a spacecraft simulation is assembled.

The execution model is built around three concepts: Processes, Tasks, and Modules. A SimBaseClass container holds one or more processes, each of which owns one or more tasks, and each task holds a list of modules that are called in order at a specified time step. Every simulation starts the same way:

# The invariant setup pattern in every Basilisk script
scSim = SimulationBaseClass.SimBaseClass()
dynProcess = scSim.CreateNewProcess("simProcess")
dynProcess.addTask(scSim.CreateNewTask("simTask", macros.sec2nano(2.0)))
# All modules are then added to this task
scSim.AddModelToTask("simTask", scObject) # spacecraft dynamics
scSim.AddModelToTask("simTask", thrusterSet) # thruster effector
scSim.AddModelToTask("simTask", sNavObject) # navigation sensor
scSim.AddModelToTask("simTask", attGuidance) # guidance law
scSim.AddModelToTask("simTask", mrpControl) # feedback controller

The time step, which is 2 seconds in the example above, is specified in nanoseconds using the macros.sec2nano() utility. Everything in Basilisk's internal clock runs in nanoseconds, which initially felt unusual, but it turns out it eliminates floating-point precision issues that would accumulate over multi-day mission simulations.

The modules communicate exclusively through typed messages. A module declares one or more output messages, which other modules subscribe to via their input message connections. There is no shared global state and information only flows through explicitly declared message subscriptions. This matters enormously for my project, because when I wrap Basilisk inside a Gymnasium environment, the "observation" that the RL agent receives will be exactly the output of the simpleNav navigation module, and the "action" it outputs will be written directly into the THRArrayOnTimeCmdMsg thruster command message. The message architecture is, conveniently, already the interface I need.

Architecture Insight

Basilisk's message-passing design has the same form as the MDP formulation I will use. The simpleNav output message is the observation. The THRArrayOnTimeCmdMsg is the action. The simulation stepping from one timestep to the next is the transition function. Building the Gymnasium wrapper will largely consist of wrapping these existing interfaces rather than reimplementing them.

03 The Thruster Example, Taken Apart

The most important script I worked through this week was scenarioOrbitManeuverTH, which simulates a full Hohmann transfer from Low Earth Orbit to Geostationary Orbit using finite thruster burns rather than the instantaneous impulsive delta-v used in earlier simpler examples. This distinction is critical for my project. Prior Basilisk orbit maneuver examples like scenarioOrbitManeuver just add a velocity vector to the spacecraft state at a specific instant, which is convenient but physically unrealistic. For RL-based control, I need the spacecraft to actually fire a thruster for a computed duration, consuming propellant, building up velocity through a force that acts over time, and with an attitude controller that keeps the spacecraft pointed in the correct burn direction throughout. This example does all of that.

3000 N Thruster 1 — GTO departure burn (perigee)
420 N Thruster 2 — circularization burn (apogee)
318 s Specific impulse (Isp) for both thrusters
RKF45 Variable-step integrator used for the finite burns

The thrusterStateEffector Module

The thruster is modeled via the thrusterStateEffector module, which is the physically correct approach for finite burns. Unlike a simple force vector added externally, the state effector formulation properly couples the thruster dynamics to the spacecraft's equations of motion. The mass changes as propellant is consumed, and this is tracked self-consistently through the integration. Each thruster is defined via the simIncludeThruster.thrusterFactory() utility, which takes a thruster model type, its location in the body frame, its pointing direction, and parameters including maximum thrust, nozzle area, and specific impulse:

# Create the thruster factory and define two thrusters
thFactory = simIncludeThruster.thrusterFactory()

# Departure burn thruster: 3000 N, high thrust for GTO insertion
thFactory.create('Blank_Thruster', [0,0,1], [0,0,1],
                 cutoffFrequency=.1,
                 MaxThrust=3000.0,
                 areaNozzle=0.046759,
                 steadyIsp=318.0)

# Circularization thruster: 420 N, lower thrust for GEO insertion
thFactory.create('Blank_Thruster', [0,0,1], [0,0,1],
                 cutoffFrequency=.1,
                 MaxThrust=420.0,
                 areaNozzle=0.046759,
                 steadyIsp=318.0)

# Attach to spacecraft so the thruster set is now a mass-varying state effector
thFactory.addToSpacecraft("GTOThrusterDynamics", thrusterSet, scObject)

The thruster is commanded through the THRArrayOnTimeCmdMsg message, which specifies how long each thruster in the array should fire. The burn duration is calculated analytically before each phase under a constant-mass assumption, which is standard practice for rough planning when the propellant fraction is small:

# Burn time under constant-mass approximation
$$t_{\text{burn}} = \frac{\Delta v \cdot M_{\text{sc}}}{F_{\text{thrust}}}$$ # From the script: first burn calculation
dv0 = v0p - v0 # required delta-v at perigee
t0Burn = dv0 / (3000.0 / scObject.hub.mHub) # burn duration (seconds)

# Write the on-time message: [thruster_1_time, thruster_2_time]
ThrOnTimeMsgData.OnTimeRequest = [t0Burn, 0]
thrOnTimeMsg.write(ThrOnTimeMsgData, time=simulationTime)

This constant-mass approximation becomes less accurate as the propellant fraction increases. For the deep-space missions I am targeting, where propellant mass can be a large fraction of total spacecraft mass, I will eventually need to use a proper finite-burn propagation that accounts for continuously changing mass through the rocket equation. Basilisk's thrusterStateEffector already does this integration internally and the approximation is only used here to estimate how long to command the burn. The simulation itself correctly propagates the mass change.

Attitude Control During the Burn

A spacecraft cannot just fire a thruster in an arbitrary direction and it has to be pointed correctly first. This example uses a full attitude control loop to keep the spacecraft aligned with the velocity vector throughout the burn, which is the correct direction for a Hohmann transfer. Five modules work together to make this happen, and understanding each of them is essential because I will need an equivalent attitude-keeping loop in my RL environment to make the commanded thrust go in the right direction.

# The full attitude control chain:
# 1. velocityPoint: computes the required attitude (reference frame aligned with velocity)
attGuidance = velocityPoint.velocityPoint()
attGuidance.transNavInMsg.subscribeTo(sNavObject.transOutMsg)
attGuidance.mu = mu

# 2. simpleNav: reads the true spacecraft state and outputs a "sensed" state
# This is the observation function for the RL agent
sNavObject = simpleNav.SimpleNav()
sNavObject.scStateInMsg.subscribeTo(scObject.scStateOutMsg)

# 3. attTrackingError: computes MRP error between commanded and current attitude
attError = attTrackingError.attTrackingError()
attError.attRefInMsg.subscribeTo(attGuidance.attRefOutMsg)
attError.attNavInMsg.subscribeTo(sNavObject.attOutMsg)

# 4. mrpFeedback: PD controller on MRP attitude error gives the torque command
mrpControl = mrpFeedback.mrpFeedback()
mrpControl.guidInMsg.subscribeTo(attError.attGuidOutMsg)
mrpControl.K = 3.5 # proportional gain
mrpControl.P = 30.0 # derivative gain

# 5. extForceTorque: applies the commanded torque to the spacecraft
extFTObject.cmdTorqueInMsg.subscribeTo(mrpControl.cmdTorqueOutMsg)

The velocityPoint module deserves special attention here. It takes the spacecraft's translational navigation state and computes a reference attitude that aligns the spacecraft with the orbit velocity frame. For my RL environment, I will need an equivalent velocity-pointing capability for thrust maneuvers. The attitude control loop runs inside Basilisk; the RL policy only needs to specify when and for how long to fire the thrusters, not how to orient itself during the burn.

Key Design Decision for the RL Wrapper

The thruster command interface in Basilisk is THRArrayOnTimeCmdMsg, which is a vector of on-times for each thruster. The RL agent's action will be exactly this: an array of burn durations, one per thruster, at each decision step. This is a bounded continuous action space, where zero means no burn and the maximum is physically constrained by the available propellant and the orbital maneuver constraints. This maps cleanly onto PPO's Gaussian policy head for continuous control.

The Variable-Timestep Integrator

One subtle but important choice in the example is the use of the svIntegratorRKF45 integrator rather than the default fixed-step RK4. The RKF45 (Runge-Kutta-Fehlberg 4th/5th order) is a variable-step integrator that adapts its step size to maintain a target local truncation error. During the coast phases, where dynamics are smooth, it can take large steps. During and immediately after a burn, when the thrust is changing the trajectory rapidly, it automatically tightens the step size to maintain accuracy.

# Set the RKF45 integrator which is critical for accurate finite burn propagation
integratorObject = svIntegrators.svIntegratorRKF45(scObject)
scObject.setIntegrator(integratorObject)
# The integrator adapts its internal step size to maintain accuracy
# during burn transients, while using large steps during coast arcs.
# This is important when thruster on-time durations are much shorter
# than the orbital period (as they are for finite deep-space burns).

For deep-space missions with long coast arcs interspersed with short burns, this variable-step capability is necessary for getting numerically accurate results without prohibitive computational cost. I had already flagged computational efficiency as a major challenge and the RKF45 integrator is one of Basilisk's tools for managing this tradeoff.

State Recovery: rv2elem and elem2rv

The orbitalMotion utility module provides the classical orbit element conversions I studied from Schaub and Junkins last week. The two most important functions are orbitalMotion.rv2elem(mu, r, v) and its inverse orbitalMotion.elem2rv(mu, oe). In the example, rv2elem is called after the first burn to compute the current orbital elements and then find the time to apogee, where the second burn will occur. This is a direct illustration of the state representation question I raised last week: the simulation propagates in Cartesian coordinates, but physically meaningful decisions (like "when is apogee?") require the orbital element representation.

# After the first burn, recover orbital elements from the Cartesian state
rVt = unitTestSupport.EigenVector3d2np(posRef.getState())
vVt = unitTestSupport.EigenVector3d2np(velRef.getState())

oe1 = orbitalMotion.rv2elem(mu, rVt, vVt)

# Use eccentric anomaly to find the remaining time to apogee
E1 = 2*np.arctan(np.sqrt((1-oe1.e)/(1+oe1.e)) * np.tan(oe1.f/2))
T3Full = np.pi / np.sqrt(mu / oe1.a**3)
T3P = np.sqrt(oe1.a**3 / mu) * (E1 - oe1.e*np.sin(E1))
T3 = macros.sec2nano(T3Full - T3P) # coast time to apogee

The RL agent will need access to this conversion. When constructing the observation vector for the policy network, I plan to augment the raw Cartesian state with the derived orbital elements, as I discussed last week. The orbitalMotion module makes this straightforward because with every step, after calling scSim.ExecuteSimulation(), I can call rv2elem on the current state and append the elements to the observation.

The orbital radius during the Hohmann transfer in scenarioOrbitManeuverTH. The initial flat region at ~7,000 km is LEO; the two burns are visible as the upward ramp during the transfer ellipse; the final plateau near GEO (~42,000 km) is the circularized target orbit.

04 Multi-Body Trajectories and Hyperbolic Arrivals

The thruster example gave me confidence in the core simulation loop, but my project requires more than Earth-orbiting maneuvers. I spent the second half of the week studying two examples that directly address the interplanetary and multi-body dynamics I will need: scenarioPatchedConics and scenarioJupiterArrival.

The scenarioPatchedConics example is particularly instructive because it is the closest existing Basilisk scenario to my eventual problem. It simulates a full Earth-to-Jupiter transfer, beginning with a hyperbolic departure from LEO, proceeding through a heliocentric Hohmann transfer ellipse, and arriving at a Jupiter flyby, all in one continuous simulation. Critically, it does this by running Basilisk's N-body integrator throughout, with Earth, Sun, and Jupiter all contributing to the gravitational acceleration simultaneously. The patched-conics model is used only to design the initial trajectory; the simulation propagates the actual dynamics with all bodies included from the start.

Basilisk always propagates the state in a single inertial frame, but the user changes which body is designated the central body for logging purposes as the spacecraft transitions between gravitational regimes. Early in the simulation, position and velocity are logged relative to Earth. After crossing the Earth's sphere of influence, the central body is switched to the Sun for the heliocentric transfer phase. As the spacecraft approaches Jupiter, the central body switches again. The code manages this by stopping the simulation, adjusting the spacecraft's state vector to account for the offset between reference frames, and restarting with a new logging frame.

# scenarioPatchedConics: frame switching at the Earth SOI boundary
# All three bodies are always contributing gravitational acceleration
# This is not patched conics in the dynamical sense, only in the initial
# design sense. The simulation integrates the full N-body problem.
gravFactory = simIncludeGravBody.gravBodyFactory()
earth = gravFactory.createEarth()
sun = gravFactory.createSun()
earth.isCentralBody = True # initial reference which switches to Sun after SOI exit

# After crossing the Earth SOI, switch the central body for logging
sun.isCentralBody = True
earth.isCentralBody = False

# Adjust spacecraft state to Sun-centered coordinates
# (add Earth's heliocentric velocity to the spacecraft's geocentric velocity)

The standard approach of always using heliocentric inertial coordinates for the state vector, and only switching to planetocentric coordinates for reward computation near flyby bodies, seems to be what Basilisk supports naturally. The message logged by simpleNav always reports state relative to the current central body, so I will need to be careful about which body is designated central at each phase when reading out the observation for the RL agent.

The scenarioJupiterArrival example covers a hyperbolic capture maneuver. The spacecraft arrives on a hyperbolic approach trajectory, propagates to periapsis, and then fires an impulsive delta-v burn to slow down into a circular parking orbit. This example demonstrated concretely how Basilisk handles hyperbolic orbits. The state initialization uses the same orbital element structure (orbitalMotion.ClassicElements()), with a negative semi-major axis indicating the hyperbolic conic. The simulation time for the first phase (initial position to periapsis) is computed from the hyperbolic time equation:

$$ N = \sqrt{\frac{\mu}{(-a)^3}}\,(t - t_p) = e\tan\zeta - \ln\!\left[\tan\!\left(\frac{\zeta}{2} + \frac{\pi}{4}\right)\right] $$

where $\zeta = \cos^{-1}\!\left(\frac{ea}{a-r}\right)$ is the hyperbolic anomaly analog to the eccentric anomaly on an ellipse. The orbitalMotion module handles these conversions, so I do not need to implement the hyperbolic time equation myself, but understanding it is essential for computing meaningful observation features (like "time to periapsis") for the RL agent during a flyby approach.

What the Examples Tell Me About Thruster Use

Neither scenarioPatchedConics nor scenarioJupiterArrival uses thrusters. Instead, they use impulsive delta-v. For interplanetary burns, the delta-v is first computed analytically (using vis-viva or the Lambert solver), and then a thruster burn of the appropriate duration replaces the instantaneous velocity change. The scenarioOrbitManeuverTH example demonstrates this translation pattern on an Earth orbit; I will need to generalize it to heliocentric and planetocentric frames across multiple mission phases.

There is also a dedicated scenarioFlybySpice example that uses real SPICE ephemeris kernels to drive the spacecraft through Venus, Earth, and Mars flybys with accurate planetary positions. This example additionally shows the velocityPoint module being used to align the spacecraft with the velocity frame during flybys and is the exact same attitude guidance module that the thruster example uses during burns. For my project, this SPICE-driven approach becomes relevant once I need to simulate actual historical mission geometries for validation purposes, though I will start with simplified fixed-planet models to keep the first iteration manageable.

05 The Monte Carlo Machinery

Monte Carlo validation is the primary means by which I will demonstrate that the trained RL policy is actually robust to realistic uncertainties, not just a policy that memorized a specific nominal trajectory. Basilisk has a built-in Monte Carlo framework, and studying it this week showed what kind of validation I should plan for.

The scenarioMonteCarloAttRW example demonstrates the complete MC setup. The entry point is the Controller class from Basilisk.utilities.MonteCarlo. The user specifies the number of runs, the random seeds, and the number of parallel threads. Then dispersions, or the statistical distributions over simulation parameters that will be sampled independently in each run, are defined:

# Basilisk Monte Carlo setup (from scenarioMonteCarloAttRW)
monteCarlo = Controller()
monteCarlo.setShowProgressBar(True)
monteCarlo.setSimulationFunction(myScenario)
monteCarlo.setExecutionCount(500) # 500 independent runs
monteCarlo.setThreadCount(4) # parallel execution

# Define dispersions as each run samples from these independently
# For trajectory work I'll adapt these to spacecraft state/control uncertainties
dispMass = UniformDispersion("scSim.scObject.hub.mHub", [712.5, 787.5])
dispInertia = NormalVectorCartDispersion("scSim.scObject.hub.IHubPntBc_B", ...)

# A retention policy specifies which messages to log across runs
retentionPolicy = RetentionPolicy()
retentionPolicy.addMessageLog(transNavOutMsgName, ["r_BN_N", "v_BN_N"])
monteCarlo.addRetentionPolicy(retentionPolicy)

# Execute and collect. Failed jobs should be zero
failures = monteCarlo.ExecuteSimulation()

For my trajectory optimization application, the dispersions I care about are directly analogous to those used in Zavoli and Federici (2021): initial state noise (position and velocity errors representing navigation uncertainty), control actuation errors (thrust magnitude and direction errors), and missed thrust events (a run where a planned burn partially or fully fails). The Basilisk MC framework can accommodate all of these through its dispersion API.

One important implementation note from the MC documentation: all simulation objects that will be varied between runs must be attached as attributes of the simulation container (scSim). If a module is only stored in a local Python variable, the MC framework cannot find it to apply the dispersion. This is a subtle but critical requirement that I will need to respect when designing my Gymnasium wrapper. The wrapper class will need to store all Basilisk modules as attributes that the MC framework can reference.

Monte Carlo & RL: Two Layers of Validation

There will be two separate uses of Monte Carlo methods in my project. The first is during training: each RL training episode is itself a kind of Monte Carlo trial, where the environment is initialized with random perturbations so the policy learns to be robust. The second is during validation: after training is complete, I run a formal MC campaign with hundreds of fixed scenarios to measure policy performance statistics. Basilisk's built-in MC Controller is designed for the second use; the first will be handled by my Gymnasium wrapper's reset function, which will sample from the dispersion distributions each time a new episode begins.

06 Mapping Basilisk to a Gymnasium Environment

Having now worked through Basilisk's architecture and the relevant examples in detail, I can state concretely what the Gymnasium wrapper needs to do. This is the central implementation task of next week, and it is worth being explicit about the design before writing any code.

A Gymnasium environment implements four methods: reset(), step(action), render(), and a definition of the observation and action spaces. The mapping to Basilisk is as follows.

# Conceptual mapping: Gymnasium ↔ Basilisk
## reset()
# → create a new SimBaseClass
# → configure spacecraft(), thrusterStateEffector(), simpleNav()
# → initialize orbital elements with randomized dispersion
# → call scSim.InitializeSimulation()
# → return initial observation from sNavObject output messages
## step(action)
# action = array of thruster on-times [t_thr0, t_thr1, ...]
# → write action to THRArrayOnTimeCmdMsg
# → call scSim.ConfigureStopTime(t_current + dt_step)
# → call scSim.ExecuteSimulation()
# → read simpleNav output messages → observation
# → call orbitalMotion.rv2elem() → augment observation with orbital elements
# → compute reward (fuel consumed, trajectory error, constraint violations)
# → check termination condition (mission complete / fuel exhausted / lost)
# → return (observation, reward, done, info)
## observation_space
# → Box(low, high): [r_x, r_y, r_z, v_x, v_y, v_z, mass, a, e, i, Omega, omega, nu]
# positions in m, velocities in m/s, mass in kg, angles in radians
## action_space
# → Box(low=0, high=max_on_time): one on-time per thruster

The biggest open question is the granularity of the RL decision step. Each call to step() advances the simulation by some time interval dt_step. If this is set to 2 seconds (the integration timestep), the RL agent makes thousands of decisions per orbit. If it is set to 1 hour, the agent makes coarser decisions but the episode is tractable in terms of total timesteps. The Zavoli and Federici approach used 40 fixed decision points across the entire Earth-Mars transfer, which amounts to roughly one decision every 9 days. I will begin with a similar coarse discretization and refine based on training stability.

— Design Question for Next Week —

Should the RL agent command a burn on-time (how long to fire this step) or a delta-v vector (what velocity change to achieve)? The first maps directly to the THRArrayOnTimeCmdMsg interface and is physically grounded. The second is more abstract but may be easier for the policy to learn because it separates the propulsion physics from the trajectory planning. Basilisk's velocityPoint module plus a built-in thrust duration calculator could serve as a lower-level controller, letting the RL agent operate at the delta-v planning level rather than the raw actuator level. I need to decide which abstraction level to start with.

My current leaning is to start at the delta-v abstraction level to simplify early training, then verify that the Basilisk thruster simulation correctly executes those burns. Once I have evidence that the policy is learning something sensible, I can lower the abstraction level toward raw thruster commands as the problem complexity increases.

07 Looking Ahead

The next two weeks are the most concrete implementation phase of the entire project. Week 4 is for finalizing the MDP formulation of state space, action space, and a preliminary reward function, along with starting the Gymnasium wrapper architecture. Week 5 is for getting the basic wrapper class functional: initialization, state observation, and episode termination logic. The baseline from which I want to depart is a working environment where a random policy can run episodes without crashing, even if it achieves nothing useful. Getting to that baseline, combined with what I learned this week about Basilisk's module structure, is the critical path.

A few specific challenges I am thinking about. The first is the cold start problem of the simulation. Every call to reset() needs to create a fresh simulation object, configure all the modules from scratch, and call InitializeSimulation(). Basilisk does not have a built-in checkpoint-restore mechanism, so there is overhead associated with each episode reset. For RL training with millions of environment steps, this overhead will accumulate. I may need to profile how long InitializeSimulation() takes for different mission configurations and decide whether to accept this cost or find a workaround.

The second challenge is the reference frame consistency issue raised by the patched conics example. During training, episodes will span different mission phases, including Earth departure, heliocentric cruise, flyby approach, and the observation representation needs to be consistent across these phases. I am leaning toward always using heliocentric inertial coordinates for the position and velocity components of the observation, and computing planetocentric elements separately only for reward computation near flyby bodies. This avoids the discontinuities that would arise from switching the primary reference frame mid-episode.

One question I would appreciate feedback on from anyone reading this: for building a Gymnasium-compatible wrapper around a simulation framework like Basilisk, is it better to run the Basilisk simulation in a separate process and communicate with it via inter-process messaging (for robustness and clean separation), or to run it in the same Python process as the RL training loop (for lower latency per step)? My current plan is same-process, but I am wondering whether it may be problematic to reset and re-run many times within the same Python session without memory leaks.
